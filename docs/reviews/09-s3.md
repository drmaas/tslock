# Review: @tslock/s3

**Spec:** `docs/specs/09-s3.md`
**Plan:** `docs/plans/09-s3.md`

## Summary

The S3 provider implements `StorageAccessor` over `@aws-sdk/client-s3` and wraps it with `StorageBasedLockProvider`. Locking uses S3 conditional writes: `PutObject` with `IfNoneMatch: "*"` for create-if-not-exists and `IfMatch: <ETag>` for compare-and-swap, both preceded by `HeadObject`. The spec and plan are thorough and well-structured. However, two correctness issues need attention before implementation: (1) `updateRecord` returns `false` on `HeadObject` 404 instead of propagating, which prevents `StorageBasedLockProvider` from clearing the registry cache — the lock gets stuck if the object is externally deleted; (2) S3 user metadata keys are lowercased by the server on read, but the spec/plan uses camelCase keys (`lockUntil`, `lockedAt`, `lockedBy`) — this could cause all metadata reads to return `undefined`. The `endpoint` field in `S3ProviderConfig` is also not used by the provider (misleading). The conditional-put approach is correct; the issues are in the error-handling and metadata-handling details.

Vision §8 "LocalStack for S3/DynamoDB" — the spec uses LocalStack for integration tests. Framework-agnostic.

## Technical Correctness

**S3 conditional writes — correct approach:**
- `PutObject` with `IfNoneMatch: "*"` — correct S3 create-if-not-exists. Supported on AWS S3 (strong consistency since 2020) and LocalStack ≥ 2.x. ✅
- `PutObject` with `IfMatch: <ETag>` — correct S3 compare-and-swap. The ETag changes on every write, so `IfMatch` ensures the object hasn't been modified since `HeadObject`. ✅
- `HeadObject` before each conditional put — correct (fetches ETag and metadata; an optimization, the conditional put is the atomic check). The spec correctly acknowledges the TOCTOU: "HeadObject is only an optimization... the conditional IfMatch/IfNoneMatch is the atomic check." ✅

**Error detection — correct:**
- `name === 'NotFound'` or `$metadata.httpStatusCode === 404` → not-found. ✅
- `name === 'PreconditionFailed'` or `412` → conditional-fail. ✅
- The plan's risk table also accepts `name === 'NoSuchKey'` (some SDK paths throw this instead of `NotFound` for `HeadObject`). Good defensive coverage. ✅

**Empty body `Body: ''`:** Correct — all state lives in user metadata. The plan mentions `ContentType: 'text/plain'` for tidiness (not required). ✅

**`Utils.toIsoString()` for metadata values:** Correct — produces ISO-8601 with 3-digit millis (e.g., `2018-12-07T12:30:37.810Z`). `Date.parse()` on read. ✅

**`insertRecord` logic:**
1. `HeadObject` → if exists, return `false`.
2. If `NotFound` → `PutObject` with `IfNoneMatch: "*"`.
3. If `PutObject` succeeds → `true`. If 412 → `false` (concurrent create). Other errors propagate.
Correct. The `IfNoneMatch` handles the race between `HeadObject` and `PutObject`. ✅

**`updateRecord` logic:**
1. `HeadObject` → if `NotFound` (404), **return `false`**.
2. Parse `lockUntil` → if > now, return `false` (still locked).
3. `PutObject` with `IfMatch: etag` → if 412, return `false`. Success → `true`.
The conditional put and expiry check are correct. **But the 404 handling is incorrect** — see Issue 1 below.

**`unlock` logic:**
1. `HeadObject` → if 404, no-op return.
2. `PutObject` with `IfMatch: etag` → if 412, swallow (best-effort). Success → void.
Correct. Best-effort unlock with `lockAtLeastFor` honored via `unlockTime(config)`. ✅

**`extend` logic:**
1. `HeadObject` → if 404, return `false`.
2. Parse `lockUntil` and `lockedBy` → if `lockedBy !== hostname`, return `false`. If `lockUntil <= now`, return `false`.
3. `PutObject` with `IfMatch: etag` → if 412, return `false`. Success → `true`.
Correct. Ownership and validity checks are in the right order. `lockedAt` and `lockedBy` preserved from existing metadata. ✅

## Gaps and Issues

1. **`updateRecord` returns `false` on `HeadObject` 404 — prevents registry cache clear.** This is the most significant issue. The spec says:
   > "If `NotFound` (404): Return `false`. (Should not happen if the registry is consistent, but handle it — the object may have been deleted externally between the registry being populated and now. The caller — `StorageBasedLockProvider.lock()` — will clear the cache on a subsequent exception path; for a clean `false` return, the lock simply appears held/missing.)"

   The problem: `StorageBasedLockProvider.lock()` only clears the `LockRecordRegistry` cache on **exception** from `updateRecord`, not on `false` (see core spec §13 algorithm: step 2 "On exception: clearCache"). Returning `false` means "lock held by someone else" — the registry keeps the name, and the next `lock()` call goes to `updateRecord` again → `HeadObject` 404 → `false` → `undefined`. The lock is **stuck forever** — it can never be re-acquired via `insertRecord` because the registry never clears.

   Compare with the Couchbase spec (`05-couchbase.md`), which correctly propagates `DocumentNotFoundError` from `updateRecord`: "Collection.get throws DocumentNotFoundError → propagate (indicates a record-registry cache miss where the record does not exist; StorageBasedLockProvider will clear the cache and retry on the next call)." The Couchbase approach triggers the cache clear and allows the next attempt to try `insertRecord`.

   **Fix:** `updateRecord` should **throw** when `HeadObject` returns 404 (not return `false`). The thrown error triggers `StorageBasedLockProvider` to clear the cache. The error propagates to the caller on this attempt (the task doesn't execute), but the next `lock()` call retries `insertRecord`. This matches the Couchbase spec's approach and is the correct ShedLock pattern. (Note: ShedLock's Java S3StorageAccessor may also return `false` here — but if it does, it has the same bug. The Couchbase spec's approach is the correct one.)

2. **S3 user metadata key casing.** AWS S3 stores user metadata keys in **lowercase** (HTTP headers are case-insensitive; S3 normalizes `x-amz-meta-LockUntil` to `x-amz-meta-lockuntil`). The AWS SDK v3 for JavaScript returns metadata keys as lowercased on read. So if you write `Metadata: { lockUntil: '...' }`, the `HeadObject` response `Metadata` would be `{ lockuntil: '...' }` — and `metadata.lockUntil` (camelCase) would be `undefined`. This would cause `parseLockUntil` to throw `LockException('Corrupted lock record: missing lockUntil')` on every read, breaking all update/unlock/extend operations.

   The spec and plan use camelCase keys throughout: `lockUntil`, `lockedAt`, `lockedBy`. The `buildMetadata` helper produces `{ lockUntil: '...', lockedAt: '...', lockedBy: '...' }`. The `parseLockUntil` helper reads `metadata?.lockUntil`. If S3 lowercases, this is `undefined`.

   **This needs verification before implementation.** The behavior may vary by:
   - AWS S3 vs S3-compatible stores (MinIO, R2, LocalStack). Some may preserve case.
   - AWS SDK v3 version. The SDK might normalize keys.

   ShedLock's Java S3 provider uses camelCase keys and works — the Java SDK v2 may preserve case, or S3 may preserve case for some paths. But for the JavaScript SDK v3, this is a known gotcha.

   **Fix options:**
   - (a) Use lowercase metadata keys: `lockuntil`, `lockedat`, `lockedby`. Ugly but safe.
   - (b) Do case-insensitive metadata key lookup on read: `metadata?.lockUntil ?? metadata?.lockuntil`.
   - (c) Verify the actual behavior of `@aws-sdk/client-s3` v3 and document the finding.

   Option (b) is the most robust — it works regardless of casing behavior. Recommend implementing a `getMetadata(metadata, key)` helper that tries the key as-is and lowercased.

3. **`endpoint` field in `S3ProviderConfig` is not used.** The spec says `endpoint` is "informational only — the user constructs their own `S3Client` (so they control region, credentials, forcePathStyle, endpoint, etc.). The provider reads `bucket` and `objectPrefix` from `S3ProviderConfig`." If the provider doesn't use `endpoint`, it shouldn't be in `S3ProviderConfig` — it's misleading. A user might set `endpoint` in `S3ProviderConfig` expecting the provider to use it, but it has no effect. **Fix:** Remove `endpoint` from `S3ProviderConfig` and `createS3ProviderConfig`. Document in the README that the user configures `endpoint` on their `S3Client` directly.

4. **`createS3ProviderConfig` factory is redundant.** It validates `bucket` non-empty and defaults `objectPrefix`. The `S3LockProvider` constructor could do this inline. The factory adds a second entry point with no additional value. Minor — not a blocker, but consider simplifying to one entry point.

5. **`insertRecord` `HeadObject` optimization adds a round-trip.** The `insertRecord` does `HeadObject` first, then `PutObject` with `IfNoneMatch: "*"` only if `HeadObject` returned 404. But `PutObject` with `IfNoneMatch: "*"` alone is sufficient — it fails with 412 if the object exists. The `HeadObject` is an optimization to avoid sending `PutObject` when the object is known to exist. But in the common case (first lock attempt on a new name), `HeadObject` returns 404, and then `PutObject` is sent — two round-trips instead of one. For the "lock already exists" case, `HeadObject` returns success, and `PutObject` is skipped — one round-trip saved. The trade-off is reasonable for the insert path (which happens once per lock name). The spec acknowledges this. ✅

6. **`extend` preserves `lockedAt` but spec's `buildMetadata` uses `config.createdAt`.** The spec's `extend` says `lockedAt: Metadata.lockedAt` (preserved from existing). The plan's `extend` code uses `head.metadata?.lockedAt ?? Utils.toIsoString(config.createdAt)`. The `??` fallback to `config.createdAt` is sensible (corrupt record defense). But the spec says `lockedAt: Metadata.lockedAt` without the fallback. Minor — the plan's defensive fallback is better than the spec's strict preservation. Recommend updating the spec to match the plan.

## Recommendations

1. **Fix `updateRecord` 404 handling.** Change `updateRecord` to **throw** when `HeadObject` returns 404, instead of returning `false`. This triggers `StorageBasedLockProvider` to clear the `LockRecordRegistry` cache, allowing the next `lock()` call to retry `insertRecord`. This matches the Couchbase spec's approach and is the correct ShedLock pattern. Update the spec's error handling table and the plan's `updateRecord` implementation and unit test accordingly (the unit test "Missing record: HeadObject 404 → returns `false`" should become "Missing record: HeadObject 404 → throws").

2. **Verify and fix S3 metadata key casing.** Before implementation, test whether `@aws-sdk/client-s3` v3 returns metadata keys lowercased on `HeadObject`. If it does: (a) use lowercase keys in `buildMetadata`, or (b) implement a case-insensitive `getMetadata(metadata, key)` helper that tries the key as-is and lowercased. The integration test against LocalStack will catch this, but it's better to address it in the unit tests first.

3. **Remove `endpoint` from `S3ProviderConfig`.** The provider doesn't use it. Document in the README that users configure `endpoint` on their `S3Client`.

4. **Update spec's `extend` to match plan's defensive fallback** for `lockedAt` (`Metadata.lockedAt ?? Utils.toIsoString(config.createdAt)`).

5. **Consider simplifying to one config entry point** — either `S3ProviderConfig` (validated in constructor) or `createS3ProviderConfig` (factory), not both.

## Verdict: NEEDS REVISION

The conditional-put approach (`IfNoneMatch`/`IfMatch`) is correct and the spec/plan structure is thorough. However, two correctness issues require fixes before implementation: (1) `updateRecord` returning `false` on `HeadObject` 404 prevents registry cache clearing — the lock gets stuck if the object is externally deleted (fix: throw on 404, matching the Couchbase spec); (2) S3 metadata key casing could cause all metadata reads to return `undefined`, breaking update/unlock/extend (fix: verify behavior and use lowercase keys or case-insensitive lookup). The `endpoint` field in `S3ProviderConfig` should also be removed (not used by the provider). With these fixes, the spec and plan are implementation-ready.
